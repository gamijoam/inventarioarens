# Sprint 2 POS performance y UX (2026-07-21)

Quick wins para bajar latencia del POS, eliminar N+1, y escalar
catalogos a >100k productos. Continua `docs/SPRINT1_POS_2026-07-21.md`.

## QW5 — Debounce + AbortController en busqueda POS

Antes: `useProducts` se llamaba en cada tecla con `staleTime: 0, refetchOnMount: 'always'`.
Eso disparaba una query por cada pulsacion sin cancelacion.

Ahora:
- Nuevo hook `usePosProductsDebounced(term, warehouseId, { debounceMs: 200 })`
  en `frontend/src/features/pos/api.ts`. Mantiene un `debouncedSearch` interno
  y solo dispara la query cuando el termino se estabiliza.
- `useProducts` (inventory-center) acepta ahora `signal` en su queryFn y
  lo pasa a `getPaginated`, que a su vez lo pasa a `api.get`. axios
  automaticamente aborta el fetch cuando TanStack Query lo descarta por
  una query mas reciente.
- `PosTerminal` consume `usePosProductsDebounced` (sin debounce manual).

Resultado: 1 fetch por rafaga de teclas, no 1 fetch por tecla.

## QW6 — Eager loads para evitar N+1

- `ProductController::index` ahora hace `with('categories.parent')` en
  lugar de `with('categories')`. El `ProductResource` accede a
  `$category->parent` y antes disparaba 1 query por categoria.
- `PosOrderController::show` agrega `payments.paymentMethod` al `load()`
  para que `PosPaymentResource::paymentMethod_name` no sea lazy.
- `SaleItemResource` ya no ejecuta `ProductUnit::query()->whereIn(id)`
  por cada item. El `PosOrderController` ahora pre-carga TODOS los
  product_units de los items en UNA sola query (`preloadSerialUnits`)
  y los expone via `Request::attributes->set('serial_units_lookup', ...)`
  (no via static del Resource, para no contaminar estado entre
  requests en procesos de larga vida como Octane).

Resultado: `GET /pos/orders?status=open` con 10 ordenes serializadas pasa
de 1 + 10*N queries a 3 queries (orders, sales, product_units).

## QW7 — Indices compuestos + pg_trgm

Migracion `2026_07_20_231930_add_performance_indexes_to_pos_and_products.php`:
- `pos_orders(tenant_id, status, opened_at DESC, id DESC)` — cubre el
  listado de pendientes con `latest('opened_at')` sin sort en memoria.
- `product_units(tenant_id, product_id, warehouse_id, status)` — cubre el
  scanner y lookup de IMEIs.
- `stock_balances(tenant_id, warehouse_id, product_id)` — cubre stock por
  warehouse.
- Trigrama en `products.name`, `products.sku`, `products.barcode` para
  busqueda fuzzy. Solo se crean si la extension `pg_trgm` esta
  instalada en el servidor; si no, la migracion continua sin abortar
  (los indices compuestos ya cubren el grueso del rendimiento).

Todos los indices usan `CREATE INDEX IF NOT EXISTS`, por lo que la
migracion es idempotente y se puede re-ejecutar.

## QW10 — `track_stock=false` en POS

Regla acordada: si `products.track_stock = false` (servicios,
suscripciones, conceptos facturables), el POS:
- NO exige stock disponible al agregar al carrito.
- NO genera movimiento de inventario al confirmar la venta.
- Igual descuenta el pago del cliente, imprime recibo y crea la CxC.

Implementacion:
- `PosCartLine` ahora expone `track_stock?: boolean` (default `true`).
- `addProduct` lo setea desde el campo `product.track_stock` (default `true`
  si el campo es null, porque el backend siempre lo serializa).
- `hasStockIssue()` ignora las lineas con `track_stock === false`.
- El backend (`PosCheckoutService`) ya respeta `track_stock` en
  `InventoryMovementService::sale()` (no genera movimiento si es
  false), ver `app/Modules/Sales/Services/SaleService.php:109-130`.

Esto elimina la inconsistencia que el reporte original del auditor
senalo entre frontend y backend.
